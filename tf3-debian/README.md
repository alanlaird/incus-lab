# tf3-debian: Debian 13 VM Lab on Incus

## What Was Created

Three Debian 13 (Trixie) virtual machines provisioned on an Incus cluster using Terraform, then configured with Ansible.

- `debian1` — 172.16.11.90  (on incus4)
- `debian2` — 172.16.11.175 (on incus1)
- `debian3` — 172.16.11.200 (on incus4)

## Architecture

| Property        | Value                              |
|-----------------|------------------------------------|
| Image           | `images:debian/13/cloud`           |
| Type            | Virtual machine                    |
| vCPUs           | 1                                  |

| Secure Boot     | Disabled (`security.secureboot=false`) |

| Ansible inventory group | `debian_vms`              |

## How to Run

### 1. Initialize Terraform

```sh
cd /Users/laird/work/incus-lab/tf3-debian
terraform init
```

### 2. Plan and Apply

```sh
terraform plan
terraform apply -auto-approve
```

If any VM's `ipv4_address` is null/pending after apply, refresh and apply again:

```sh
terraform refresh
terraform apply -auto-approve
```

### 3. Manual Bootstrap (required for Debian cloud images)

The Debian 13 cloud image ships with:
- `openssh-server` NOT installed
- Root account locked (`*` in `/etc/shadow`)

Bootstrap all three VMs via `incus exec`:

```sh
PUB=$(cat debian_id_ed25519.pub)
for vm in debian1 debian2 debian3; do
  # Install openssh-server (use DEBIAN_FRONTEND=noninteractive to avoid dialog prompts)
  incus exec cluster:$vm -- env DEBIAN_FRONTEND=noninteractive apt-get install -y openssh-server

  # Inject SSH key
  incus exec cluster:$vm -- bash -c "
    mkdir -p /root/.ssh &&
    echo '$PUB' > /root/.ssh/authorized_keys &&
    chmod 700 /root/.ssh && chmod 600 /root/.ssh/authorized_keys
  "

  # Unlock root account (cloud images lock root by default)
  incus exec cluster:$vm -- passwd -d root

  # Ensure SSH is running
  incus exec cluster:$vm -- systemctl enable ssh
  incus exec cluster:$vm -- systemctl start ssh
done
```

> Note: During the initial apply run, the openssh-server postinst may block on an interactive
> dialog about `/etc/ssh/sshd_config`. If this happens, kill the `dialog` process on the VM
> and dpkg will finish automatically. On subsequent runs, use `DEBIAN_FRONTEND=noninteractive`
> to avoid this.

### 4. Run Ansible Playbook

```sh
/Users/laird/Sync/ivenv/bin/ansible-playbook -i ansible/inventory.ini ansible/playbook.yml
```

## Key Differences vs tf3-alpine

| Aspect               | tf3-alpine                          | tf3-debian                                |
|----------------------|-------------------------------------|-------------------------------------------|
| Image                | `images:alpine/3.21`                | `images:debian/13/cloud`                  |
| SSH pre-installed    | No (Alpine minimal)                 | No (Debian cloud — openssh not included)  |
| Python pre-installed | No                                  | Yes (python3.13 via cloud-init base)      |
| cloud-init support   | Limited (no cloud suffix)           | Full cloud-init via `/cloud` image        |
| Root account         | Enabled by default                  | Locked (`*`) — must run `passwd -d root`  |
| Ansible bootstrap    | Raw SSH tasks to install Python/SSH | apt module directly; SSH needs manual install |
| Package manager      | `apk` (Alpine package keeper)       | `apt` (Debian)                            |
| SSH resource name    | `tls_private_key.alpine_ssh`        | `tls_private_key.debian_ssh`              |
| Key files            | `alpine_id_ed25519[.pub]`           | `debian_id_ed25519[.pub]`                 |
| Instance resource    | `incus_instance.alpine`             | `incus_instance.debian`                   |
| Inventory group      | `alpine_vms`                        | `debian_vms`                              |
| cloud-init directive | `users:` only                       | `disable_root: false` + `users:`          |

## Teardown

To destroy only the Debian VMs (the Alpine VMs in tf3-alpine are managed by a separate state):

```sh
cd /Users/laird/work/incus-lab/tf3-debian
terraform destroy -auto-approve
```

This will delete `debian1`, `debian2`, `debian3` and remove the generated key files and Ansible inventory. It does NOT affect `alpine1`, `alpine2`, `alpine3`.
